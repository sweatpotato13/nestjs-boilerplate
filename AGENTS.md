# AGENTS.md

## Repository Purpose

- This is a NestJS boilerplate, not a product repository; keep changes reusable as a starting point for new projects and avoid adding domain-specific business logic unless requested.
- Preserve the existing module/CQRS conventions so projects cloned from this repo start with the same architecture.

## Commands

- Use `pnpm`; `pnpm-lock.yaml`, README, Husky, and CI all use pnpm. `Dockerfile` still references `yarn.lock`, which is not present, so do not assume Docker builds are green without reconciling that.
- Initial setup from README: `pnpm install`, then `pnpm prisma:migrate`, then `pnpm prisma:generate`.
- Build: `pnpm build`; Nest uses the SWC builder from `nest-cli.json` and `.swcrc`.
- Lint/format: `pnpm lint` auto-fixes TS files; pre-commit runs `pnpm lint-staged`, which applies Prettier, `pnpm lint`, and `pnpm prisma:format` to TS/JS files.
- Tests: `pnpm test`; single spec: `pnpm test -- --runTestsByPath src/path/file.spec.ts`; coverage: `pnpm test:cov`.
- API artifacts: `pnpm swagger` regenerates `public/swagger.json`; `pnpm sdk` writes the Nestia SDK to `src/api`.
- Prisma schema changes: edit `prisma/schema.prisma`, run `pnpm prisma:format`, `pnpm prisma:migrate` when the DB schema changes, and `pnpm prisma:generate` to refresh Prisma Client, `src/shared/entities`, and `prisma/README.md`.

## Architecture

- Entry points are `src/main.ts` and `src/app.module.ts`; only modules imported by `AppModule` are exposed. `TemplateModule` is a scaffold and is not currently imported.
- Feature modules live under `src/modules/{feature}` with `app/` for controllers/services, `domain/` for commands, queries, DTOs, and interfaces, `infrastructures/` for adapter/repository wiring, and `{feature}.module.ts` for Nest registration.
- Business logic uses `@nestjs/cqrs`: writes are Commands, reads are Queries, and controllers call services that dispatch `CommandBus`/`QueryBus` to handlers.
- Keep services thin; put validation, transactions, Prisma calls, and response construction in command/query handlers.
- Each feature module registers `CqrsModule`, string DI tokens such as `"UserService"` and `"PrismaService"`, and spreads `CommandHandlers`/`QueryHandlers` from handler barrel files.
- `src/common` contains cross-cutting Nest primitives: decorators, filters, guards, interceptors, and pipes.
- `src/config` loads `.env` from the repo root and provides env-backed configs for JWT, Google OAuth, Redis, Kafka, Elasticsearch, Mongoose, Mailer, and Winston.
- `src/shared` contains generated Prisma entity classes (`entities/`), base DTOs, custom HTTP errors, reusable services, and wrapper modules for JWT, Passport Google OAuth, Redis, Kafka, Elasticsearch, Mongoose, and Mailer.

## Libraries And Generated Artifacts

- Core stack: NestJS 11, TypeScript strict mode, `@nestjs/cqrs`, Prisma/PostgreSQL, Jest/ts-jest, Nestia/typia, ESLint flat config, Prettier, Winston, Passport Google OAuth, ioredis, kafkajs, Elasticsearch, Mongoose, and `@nestjs-modules/mailer`.
- `tsconfig.json` defines aliases: `@src/*`, `@common/*`, `@shared/*`, `@config`, and `@modules/*`; use them instead of long relative paths when matching existing code.
- Nestia/typia TypeScript transformers are configured in `tsconfig.json`; after installing dependencies, `prepare` runs `ts-patch install && typia patch && husky`.
- `public/swagger.json` is read at runtime by `src/main.ts`; controller/DTO changes that affect API shape should be followed by `pnpm swagger`.
- `src/shared/entities` is generated by `prisma-class-generator`; do not hand-edit generated entity files unless changing the generator output intentionally.

## Conventions

- Write new code and documentation comments in English; existing Cursor rules require English comments and API documentation.
- Controller methods should include Nestia-readable JSDoc such as `@tag`, and authenticated routes use `@security bearer` in existing controllers.
- DTO responses usually expose a static `of(...)` factory and use `class-validator` decorators; prefer returning DTOs over raw Prisma entities from handlers.
- Use custom HTTP exceptions from `src/shared/models/error/http.error.ts` for application errors; these log through the shared Winston logger.
- Keep `.env` local and ignored; update `.env.example` when adding required environment variables.

## Local Services

- `docker-compose.yml` defines Postgres, Hasura, MongoDB, mongo-express, Redis, redis-commander, Elasticsearch, Kibana, Kafka/Zookeeper, and the Nest container.
- Default app port is `8000`; Swagger UI is mounted at `/api-doc` from the generated `public/swagger.json`.
- CI runs on Node `20.x` and `22.x` and verifies install/build, Prisma generate plus tests, and lint.
